## 1. 가상 진열장(My Bar) 고도화 및 정렬 보존 (완료)

### 1.1 글로벌 상태 보존 (Zustand 마이그레이션)
* **현상**: 기존에는 선반 위의 술병 정렬 상태가 `HomeScreen` 컴포넌트의 로컬 상태(`useState`)로만 묶여 있어, 탭 메뉴를 이동했다가 복귀하면 사용자가 힘껏 드래그하여 배치한 술병들의 순서가 정적인 기본 상태로 매번 되돌아가는 문제가 있었습니다.
* **해결책**: `shelf0Ids` 및 `shelf1Ids` 정렬 리스트 상태를 Zustand 전역 스토어인 `useAppStore` 내부로 승격시켜 영구 보존하고, 화면 전환 후 돌아와도 유저가 배치한 순서가 유지되도록 구현했습니다.

### 1.2 동적 인벤토리화 및 새 스피릿 영구 추가 기능
* **현상**: 기존에는 `INVENTORY` 배열이 소스 코드 내부의 상수로 고정되어 가상 바에 신규 주류를 구매해 들여놓거나 다 마신 병을 버리는 CRUD를 반영하지 못했습니다.
* **해결책**:
  * static 상수 `INVENTORY`를 완전히 제거하고 Zustand 스토어 내 동적 상태인 `inventoryItems`로 이관했습니다. 스토어 액션으로 `addInventoryItem`, `removeInventoryItem`, `updateInventoryItem`을 설계했습니다.
  * 우측 하단에 미려한 유리 질감의 Floating Action Button(FAB, `+ Add Spirit`)을 배치하고, 선택 시 **TheCocktailDB API**를 실시간 호출해 연동하는 주류 추가 모달(`AddSpiritModal`)을 제작하여 가상 바 주류 편집 기능을 완성했습니다.

### 1.3 BottomPanel 상세 편집 기능 구현
* 진열장 내 술병을 터치하면 열리는 하단 상세 정보 패널(`BottomPanel`)에서 다음의 정보를 실시간으로 수정해 글로벌 스토어에 동기화할 수 있도록 구현했습니다:
  * **별점(Rating)**: 터치로 0~5점 조절 및 저장.
  * **남은 용량(Quantity)**: Full / High / Medium / Low 용량 칩 터치 시 선반의 액체 렌더링에 실시간 동기화.
  * **메모(My Note)**: 다국어(En/Ko) 메모 인풋 편집 지원.
  * **삭제 버튼(Remove)**: 해당 주류를 진열장과 전역 인벤토리에서 안전하게 삭제.

## 2. 커스텀 믹싱 시뮬레이터 개편 (완료)

### 2.1 스토어 데이터 영속성 (Zustand Persist) 및 부활 방지
* Zustand 스토어에 `persist` 미들웨어를 입혀 `inventoryItems`, `customRecipes`, `savedRecipeIds`, `journalEntries` 등 핵심 상태들을 로컬 저장소(AsyncStorage)에 영속화했습니다.
* 최초 기동 시에만 3대 시드(Bourbon, Gin, Rum)를 주입하며, 사용자가 선반을 완전히 비운 뒤 재기동하더라도 시드가 다시 오작동하여 부활하는 현상을 막기 위해 `seedInjected` 플래그를 영속화하여 관리합니다.

### 2.2 드래그리스 탭-투-푸어 (Tap-to-Pour) 인터페이스
* 스크롤 제스처와의 마찰을 차단하기 위해 드래그 드롭 모델을 버리고, 병 탭 시 카운트(Parts)가 1씩 누적되는 직관적인 탭 계량 모델로 오버홀했습니다.
* **잔별 가이드라인 (Envelope)**: Rocks, Highball, Coupe, Shot 잔 각각의 권장 용량(min/max parts)과 ABV 가이드를 정의하여 범위 초과 투입 시 진입을 통제합니다.
* **Zustand 파생 셀렉터**: 스토어 내부 상태에는 `{ ingredientId, parts }[]` 원시 배열만 얇게 유지하고, 뷰단에서는 파생 셀렉터 훅인 [useSelectedIngredients](file:///d:/Workspace/OntheRock/OntheRock/App.tsx#L949-L974)을 활용해 객체를 실시간 조립하여 결합도를 낮췄습니다.

### 2.3 실시간 밸런스 코칭 UI
* **코칭 게이지**: 재료 푸어링 도중에 현재 믹싱 상태의 실시간 도수(ABV) 및 단맛/산미 비율을 분석하여 전용 게이지 UI로 시각화합니다.
* **교정형 힌트**: 밸런스를 벗어날 때 "단맛 과다: 레몬/라임 주스를 더해보세요" 등 비난 없는 톤의 피드백을 실시간 제공합니다.
* **Remix 기능**: 완성 서빙 화면에서 "다시 만들기" 선택 시, 넣었던 재료와 파트수를 유지한 채 다시 Pour 단계로 돌아가 믹싱 구성을 수정할 수 있습니다.

### 2.4 맛 예측 및 시음평 엔진 고도화
* **Parts 가중 평균 및 희석 연산**: 단순 평균 맛 연산을 각 재료의 parts 비중 가중 평균으로 계산하고, 얼음 갯수에 따른 희석 공식 `Dilution = 1 / (1 + iceCount * 0.08)`을 최종 도수 및 5축 맛 감도 연산에 적용했습니다.
* **시음평 작문기 [generateTastingNote](file:///d:/Workspace/OntheRock/OntheRock/App.tsx#L2476-L2508)**: 5축 맛 수치 구간(Band)에 따라 드라이함/감미로움/상큼함 등 어휘를 조합하여 매끄러운 한글/영어 시음평 문장을 동적 자동 생성합니다.
* **저널 평점 선택**: 저널 저장 시 유저가 직접 1~5점의 별점을 터치하여 보관할 수 있게 UI를 추가했습니다.

### 2.5 코사인-맨해튼 결합 유사도 및 개인화 대조군
* **유사도 공식 개선 [calcFlavorSim](file:///d:/Workspace/OntheRock/OntheRock/App.tsx#L2457-L2474)**: 맛의 방향을 측정하는 **코사인 유사도(70%)**와 맛의 강도 일치율을 측정하는 **맨해튼 거리의 역수(30%)**를 결합하여 정확도를 극대화했습니다.
* **개인화 대조군 우선순위**: 사용자가 평가(Rating)를 부여한 인벤토리 술병 및 직접 기록한 저널 목록을 1, 2순위 대조군으로 삼아, 기성 레시피보다 내 취향의 맛이 유사도 랭킹 1위로 먼저 도출되도록 대조군 매칭 엔진을 바인딩했습니다.

### 2.6 아카이브 및 탐색기 통합 필터
* **레시피 저장**: 믹싱 완료 시 사용한 재료 비율과 잔 정보를 결합하여 단계별 제조 가이드라인 텍스트 및 레시피 객체를 빌드해 저장합니다.
* **통합 검색**: 검색 스크린 및 프로필 내 저장 목록 아카이브에서 기성 레시피 외에 사용자가 직접 커스텀한 레시피(`customRecipes`)가 완벽하게 결합된 단일 통합 리스트로 노출 및 필터링됩니다.

## 3. Gemini 3.5 Flash & 이중 Firebase App 연동 (완료)

### 3.1 전용 API 키 및 직접 HTTP 통신 연동
* **상황**: Firebase SDK (`firebase/ai`)를 사용해 클라이언트 앱에서 호출할 경우, Firebase 백엔드가 프로젝트 설정(`projectId`, `appId`)과 API 키 간의 결합 관계를 강제 검증합니다. 이로 인해 메인 Firebase 환경과 별도로 생성된 외부 Gemini 전용 키(`AQ.`)를 섞어 쓰는 방식은 클라이언트 측 SDK 통과 단계에서 차단되는 제약이 있었습니다.
* **해결책**:
  * Firebase SDK를 우회하여 React Native 내장 `fetch` API를 통해 **Gemini Developer API(`https://generativelanguage.googleapis.com/...`)를 직접 호출**하도록 아키텍처를 변경했습니다.
  * [App.tsx](file:///d:/Workspace/OntheRock_new/App.tsx) 내부의 `fetchAISommelier` 함수에서 사용자의 Gemini 활성화 API 키(`AQ.Ab8RN6L...`)를 실시간 HTTP POST 요청으로 바인딩하여 백엔드 호출을 완료합니다.
  * 이로써 팀원의 메인 Firebase App 구성([src/config/firebase.ts](file:///d:/Workspace/OntheRock_new/src/config/firebase.ts))을 전혀 훼손하지 않으면서도, AI 기능만을 독립적으로 무료 요금제의 Gemini 모델에 완벽히 격리 연동시켰습니다.


### 3.2 로컬 예측 즉시 표시 + AI 소믈리에 비동기 보강 모델
* 사용자 경험 극대화를 위해 분석 버튼 클릭 시, 1500ms의 인위적 딜레이(AI 분석 연출용) 후 **로컬 맛 분석과 감도 분석을 즉시 렌더링**합니다.
* 동시에 백엔드에서 `gemini-3.5-flash` 모델을 비동기로 호출합니다. AI 결과가 성공적으로 수신되면, 하단에 골드 톤 테두리의 **"AI 소믈리에" 섹션**이 자연스럽게 추가 노출됩니다.
* 만약 네트워크 단절이나 API 제한 등으로 AI 호출이 실패하더라도, 에러를 UI로 뿜어내지 않고 로컬 예측 결과만 깔끔하게 유지하여 완벽한 오프라인/폴백 안정성을 제공합니다.

### 3.3 맛 프로필 개인화 (Taste Genome) 프롬프트 바인딩
* AI 모델 호출 시 유저의 기존 저널 평점과 인벤토리 데이터를 가공하여 도출한 **Taste Genome(취향 DNA) 데이터**를 프롬프트에 동적 삽입합니다.
* Gemini는 이를 인지하여 사용자의 취향과 비교해 "당신의 평소 취향에 잘 맞는 묵직하고 달콤한 칵테일입니다"와 같은 초개인화된 테이스팅 노트 평을 생성합니다.

### 3.4 AI 소믈리에 UI 구성
* **네임 제안 (Name Ideas)**: 입력된 비율과 글라스, 맛 프로필을 고려한 3가지 크리에이티브한 이름 제안.
* **푸드 페어링 (Food Pairing)**: 맛과 도수에 가장 어울리는 추천 안주/디저트 3선 제시.
* **개선 팁 (Improvement Tip)**: 밸런스를 개선할 수 있는 레시피 보정 팁 제공.
